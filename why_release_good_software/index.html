About me / Research / Software

I spend a significant fraction of my time developing scientific software or using codes built by other people. Over the years, I came to the realization that are compelling reasons to write good scientific software and release it publicly. I am going to list them here, and take this as a personal manifesto of how I would like to produce software for science.

Publicly release

  • It is an incentive to produce high-quality software and research
  • You get a lot of free debugging (by the people that use your code)
  • Other people can implement bug fixes/features.
  • It empowers other people to do things they could not do otherwise

Good software

  • In the long run, it saves your time

  • It saves hundreds of person-hours (And bad quality software wastes hundreds of person-hours)

  • Attracts contributors and you might get features you wanted to include but never found the time..

  • Tests make you feel better

  • It turns out that the more you do it, the less time it takes.

  • Builds trust

Reasons not to publicly release good scientific software

I can see one good reason not to publicly release good scientific software: credit recognition.

  • To the eyes of many, software does not matter: only papers and citations do. This is especially critical for early-career scientists.
  • Building a code takes a lot of effort, it would be a shame if other people were to obtain important results off your work.

(This can be solved with a proprietary time, but this would also nullify some of the advantages of public release)

I don’t see time or skills as a good reason: we expect physicists to know mathematics and to take their time to do their mathematical manipulations well. In the same spirit, we should expect scientists that write any kind of software to do it well.

Tips and tricks

  • If you want your code to be maintainable, value simplicity and readability over everything else. Yes, there are more elegant solutions, but those will be harder for others to understand.

  • Value documentation more than code. A piece of code that is obscure is not used by many people, and it destined to be forgotten. There are four different types of documentations you have to take care of: technical reference, tutorials, examples/how-tos, and overall design.

  • Choose mainstream technologies. The pool of scientists interested in your software is probably small. If you use technologies that are niche, how many of them do you think would contribute to your software? (e.g., in astronomy use Python instead of Clojure)

  • Use verbose names for variables and functions. Name a function read_ts requires the reader to figure out what ts is, but read_timeseries is immediately clear.

  • Release libraries, not scripts.

  • Use technologies that can be used by a person with a reasonable access to resources (e.g., not everyone has access to proprietary software)

~Accelerating science in the era of Open Source Software~ -A manifesto for better science-

Open source has a snowball effect that dramatically accelerates innovation.

Current system: code is developed with the only goal of answering a specific scientific question

Proposed system: code is developed with the goal of answering a specific scientific question and to provide something useful tool to the community

The only problem is credit and recognition.

This manifesto advocates for two things:

  1. Scientists should Create software with the specific intent to create something useful for the community
  2. Scientists should recognize that this kind of software development is a critical contribution to science, in the same way publishing papers is
